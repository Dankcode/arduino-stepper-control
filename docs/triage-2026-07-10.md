# Triage 2026-07-10 — regressions from commits 55e816a / 844dde1

Three reported problems, three root causes. The follow-up code now removes the OpenCV
streaming dependency and restores V2 click selection.

## 1. Manual instructions stopped working — FIXED (backend.py)

**Root cause: firmware/protocol mismatch, not the commits' logic.** The backend's
`SerialLink.command(expect_done=True)` waits for a line starting `OK:done` (v2 firmware
reply). The Arduino is still flashed with **v1** (`firmware/stepper_controller`), which
replies `OK:move_complete` — so every X/Y move waited 30 s, then returned
"Timed out waiting for move completion". The `?` handshake was fine (v1 implements it).

**Fix applied:** `command()` now accepts `OK:move_complete` and `OK:done` as completion,
and no longer fails when v1 stays silent during a long blocking move (empty first read →
keep waiting until the deadline instead of erroring immediately).

**Follow-up (your call, no urgency):** flashing `stepper_controller_v2` is still the plan —
it adds mid-move abort and acceleration. The backend now works with either firmware, so
flash whenever convenient and verify with `?` in a serial monitor first.

## 2. Routine designer forcing drag selection — FIXED

`55e816a` made RoutineDesignerV2 the default tab. Its plate canvas only supports
marquee/drag selection — no single-click select/toggle — which is the "forcing us to
drag" complaint.

**Fix applied:** RoutineDesignerV2 now supports single click, click-to-deselect,
Ctrl/Cmd+click add/remove, Shift+click rectangular range from the anchor, and drag
ranges. V2 is default again; the old RoutineBuilder is available at `?legacy=1`.

## 3. Camera stream "backend incorrect" — PARTLY FIXED

**Root cause:** the stream route needs `picamera2` importable inside the Flask process.
The earlier `cv2` dependency has been removed; streaming now uses picamera2's
`MJPEGEncoder` + `FileOutput`. If the backend venv was created without
`--system-site-packages`, it still cannot import apt-installed `picamera2`.

**Fix to run on the Pi (one time):**
```bash
sudo apt install -y python3-picamera2
python3 -m venv --system-site-packages ~/stepper-agent-venv --clear
~/stepper-agent-venv/bin/pip install -r pi_agent/requirements.txt
# restart the backend service
```

**Fix applied:** `/api/camera/status` checks the exact picamera2 encoder imports and
reports whether a camera is detected; `/api/camera/stream` starts the encoder before
returning the response, so startup failures are returned as JSON instead of a blank stream.
The camera helper no longer reports mock captures as successes. OpenCV is not required.

Note: `/api/camera/take-picture` still spawns `camera.py` as a subprocess that opens its
own `Picamera2` — if the stream is running, the sensor is busy and capture fails. That's
the CameraManager consolidation in plan v3 §3; until then, stop the stream before capturing.

## 3a. Routine failure/abort reporting — FIXED

Routine light and camera subprocess failures now stop the routine and are returned in
`/api/routine/progress.error`; they no longer look like a completed capture. Abort is now
reported as a request until the runner exits, rather than claiming it stopped while a
firmware-v1 blocking move is still underway. Connection/disconnection changes are also
blocked while a routine owns the controller.

## 6. "Where is the backend?" + routine portability (third report)

**Where the backend lives.** The backend is `src/app/RaspiBackend/backend.py` (Flask).
The dashboard talks to whatever URL is in `.env.local` — currently
`http://192.168.1.43:5000`, i.e. it expects the backend to be **running on the Pi** at
that address. All saved routines live in a SQLite file **on the machine running the
backend** (`$STEPPER_PI_HOME/routine_data.db`, default `/home/dank/routine_data.db` on
the Pi). If nothing is listening at that URL — or an old copy of backend.py is — every
save fails. Check by opening `http://192.168.1.43:5000/` in a browser: current code
answers with `"version": "2026-07-10.6"`.

The `.sql` suffix shown in the legacy routine list is a compatibility label, not a file
on disk. The canonical storage is SQLite; a successful save now reports its well count
and `storage: "sqlite"` in the API response.

**Legacy database migration.** Older Pi databases have a `routine_schedule` table
without `repeat_interval` and `repeat_count`. That caused `/routines/detail` to return
HTTP 500 and `/save_routine_sql` to roll back. `backend.py` now adds those columns in
place at startup, keeps existing routine rows, and returns the saved filename/well count
in the save response. Deploy and restart the Pi backend once to apply the migration.

**Run the backend anywhere (new).** `scripts/run_backend_local.sh` starts the backend on
this computer with data in `~/stepper-data/`. Point `.env.local` at
`http://localhost:5000`, restart `npm run dev`, and uploads work with no Pi involved
(hardware routes report unavailable, which is fine for routine authoring).

**Moving routines around (new).** Routines are now portable JSON files in the previous
uploader format `{ filename, well_data: [...], schedule }`. The importer also accepts the
legacy top-level `startTime`, `repeatInterval`, and `repeatCount` fields:
- Designer toolbar: **Export** downloads the current routine as `<name>.json`.
- Pi Routines tab: **Download** button on a selected routine fetches it from the backend
  as the same JSON.
- Both files re-import via the designer's **Import** button (and the legacy builder's
  JSON upload) — so you can author locally, export, and import against the Pi backend,
  or vice versa.

**Clearer failures.** A save that can't reach the backend now says "Could not reach the
backend at <url>" instead of a generic error, and points at the local runner script.

## 7. Motors read null bytes / moves time out (fourth report) — FIXED

**Symptom:** `Timed out waiting for move completion after '\x00\x00\x00...'`; connect
appears to succeed but no motor command works; only GPIO features (blue light) respond.

**Root cause: baud rate mismatch.** Firmware v1 (what's flashed) runs at **9600**.
Earlier releases defaulted `config.py` to **115200** (chosen for firmware v2). The Pi's
systemd unit loads `/home/dank/stepper-agent.env` only if it exists — without it the
backend opened the port at 115200 and every byte read back was garbage (`\x00...`). A
second bug compounded it:
`connect()` accepted any non-empty handshake reply, so the garbage connection passed as
"Connected".

**Fixes applied (backend.py, version 2026-07-10.5):**
- `config.py` now defaults to 9600 for the deployed v1 firmware; v2 at 115200 remains
  supported by the automatic probe.
- `connect()` now validates the `?` handshake (`OK:`/`POS` prefix required) and
  auto-detects between 9600 and 115200 — tries the configured baud, falls back to the
  other, reports which one worked. Verified with a mocked serial device: configured
  115200 against a 9600 firmware → connects at 9600 and moves complete.
- `_readline()` strips null/replacement bytes so line noise can never masquerade as a
  real reply.
- `/api/status` and `/api/connect` return the active baud rate, so the dashboard/API
  can confirm whether 9600 or 115200 was selected; the connection badge shows it.
- `.env.example` documents which firmware uses which baud.

**On the Pi (recommended):** create `/home/dank/stepper-agent.env` containing
`STEPPER_BAUD_RATE=9600` (matching v1) so the first connect attempt succeeds without
the fallback's extra ~4 s. When you flash firmware v2, change it to 115200 — or leave
auto-detect to handle it.

## 8. Motors still time out with '\x00...' (fifth report) — REVERTED TO LEGACY + PROOF NEEDED

**Critical observation: the Pi is not running the current backend.** The reported error
contains raw `\x00` bytes, but since version 2026-07-10.4 the backend strips null bytes
before any error string is built — the current code **cannot produce that message**. It is
coming from an older backend copy still running at 192.168.1.43:5000. No repo change takes
effect until a deploy lands:

```bash
PI_HOST=192.168.1.43 PI_USER=dank scripts/deploy_pi.sh
ssh dank@192.168.1.43 'sudo systemctl restart stepper-agent.service'
# verify: open http://192.168.1.43:5000/  ->  must say "version": "2026-07-10.6"
```

**Revert applied (as requested), version 2026-07-10.6.** Motor control now behaves exactly
like the previous working build (`stepperbotBackend.py` semantics):
- `SerialLink.fire()`: manual motor/steps commands are **fire-and-forget** — write the
  command, brief pause, return whatever the firmware said (or "Command sent." while it's
  still moving). No waiting for `OK:done`/`OK:move_complete`, so manual moves can never
  produce a completion timeout.
- `connect()` never refuses: it still probes 9600/115200 with a `?` handshake, but if
  nothing valid answers it connects anyway at the configured baud in **legacy mode**
  (the old build never handshook). Default baud is 9600 (v1 firmware), matching the
  previous build.
- Routine execution still waits for completion lines (it must, to sequence moves), at the
  baud the connection established.

**Definitive hardware check (new): `scripts/serial_probe.py`.** Copy it to the Pi and run
`python3 serial_probe.py /dev/ttyUSB0` with the backend stopped. It talks straight to the
board at 9600/115200/57600/74880 and hex-dumps every reply. Readable `OK:` text at some
baud = use that baud. Nothing but `00`/`FF` at every baud = the problem is the flashed
sketch, cable, or board — not the backend, and no software revert will move the motors.
