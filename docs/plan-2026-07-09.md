### New findings (this audit) — these are the "fluidity" blockers
- **N1 — routine.py was never migrated.** Cron → `cronjob.py` → `subprocess routine.py`, which still speaks the v1 blocking protocol and ignores the entire motion stack. Everything built on 07-03 is dormant in the real execution path.
- **N2 — Serial port contention.** `backend.py` holds the Arduino port persistently; `routine.py` (spawned by cron) opens the *same* port. Whichever grabs it second fails, or the backend's connection is corrupted mid-routine. This is an architecture bug, not a code bug.
- **N3 — Progress is never written.** `routine_progress` is an in-memory dict in the Flask process; `routine.py` runs in a *different process* and never updates it. `/api/routine/progress` always returns zeros, so the V2 ProgressBar polls dead data.
- **N4 — No abort path.** Firmware v2 supports `!` (abort+flush) but no backend route exposes it and no UI has a Stop button. A running routine cannot be stopped — the single biggest "not fluid" complaint.
- **N5 — V2 designer not mounted.** `page.js` still imports `RoutineBuilder`; `RoutineDesignerV2` is orphaned.
- **N6 — Styling unmigrated.** All 7 old components + `page.js` still carry one `<style jsx global>` block each (~250 hardcoded hex values total); tokens/Tailwind used only by V2 designer.
- **N7 — One global `loading` flag in ManualControl** disables every button during any action; feedback is inline `<div>` messages, not toasts.

## 2. ARCHITECTURE DECISION — single serial owner

**Root cause of N1–N4:** two processes competing for one serial port and no shared state.

**Decision: `backend.py` becomes the only process that ever touches the Arduino.**
Routine execution moves into the backend as a background thread. Cron stops spawning
`routine.py` and instead calls an HTTP endpoint.

Why this over alternatives:
- *Keep subprocess + lock file*: still can't share progress or abort without IPC; rejected.
- *Separate daemon owning serial + backend proxies to it*: correct but over-engineered for a Pi Zero; revisit only if the Flask thread proves unreliable.

Consequences: N2 gone (one owner), N3 gone (progress updated in-process), N4 trivial
(`!` sent from the request thread while the worker thread runs), and the routine runner
finally sits next to the motion modules it should use.

## 3. BACKEND ORCHESTRATION (`src/app/RaspiBackend/`)

### backend.py — add
| Function / route | Signature & behavior |
|---|---|
| `POST /api/routine/run` | body `{filename, plate?}` → validate routine exists in SQLite, refuse if `routine_runner.running`, start worker thread, return `202 {started: true}` |
| `POST /api/routine/abort` | set abort flag, send `!` via `serial_link`, wait for flush ack, mark progress `{running: false, aborted: true}` |
| `RoutineRunner` class (module-level singleton) | fields: `thread`, `abort_event`, owns writes to `routine_progress`. Methods below. |
| `RoutineRunner.run(filename, plate)` | loads well data (reuse the SQL from routine.py's `fetch_wells_to_scan`, but return full params, not just lightTime), calls `trajectory.plan_routine()` for serpentine ordering, iterates planned moves |
| `RoutineRunner._execute_step(move)` | stream `M dx dy dz` via `serial_link.command(expect_done=True)`; on `ERR:` → abort routine, log, set `progress.error`; check `abort_event` between wells |
| `RoutineRunner._capture(well_id)` | blue light (ms→s conversion HERE, once: `sleep_s = light_ms / 1000`), then camera capture; update `routine_progress = {running, wells_done, wells_total, current_well}` after each well |
| `serial_link` thread lock | add `threading.Lock` around `command()` — manual-control requests must not interleave with the runner's streamed moves; manual motor routes return `409 busy` while a routine runs |

### backend.py — change
- `api_motor(action)`: when `RoutineRunner.running` → `409 {"error": "routine running"}`.
- `api_status`: include `{"routine_running": bool, "port": SERIAL_PORT, "firmware": reply_from_"?"}`.

### routine.py — repurpose, then shrink
- Delete `StepperController` class entirely (superseded by `SerialLink`).
- Delete `execute_96well_plate_routine` (superseded by `RoutineRunner.run`).
- Keep only: `fetch_wells_to_scan` (move into backend.py or a shared `db.py`), `position_from_well_id` / `well_id_from_position` (move into `motion/kinematics.py` — they're geometry).
- End state: **delete routine.py**; CLI fallback if you want one: `python backend.py --run-routine <name>` calling `RoutineRunner` directly.

### cronjob.py — change
- `execute_routine_script()`: replace `subprocess.run([... routine.py ...])` with `requests.post(f"http://127.0.0.1:5000/api/routine/run", json={"filename": name}, timeout=5)`. Poll `/api/routine/progress` or just fire-and-forget + log.
- Add `requests` to `pi_agent/requirements.txt`.

### b_light.py — change
- Document/enforce: `automate` takes **seconds** (float). All ms→s conversion happens in `RoutineRunner._capture`. Add a guard: reject durations > 60 s (catches future unit regressions at the hardware boundary).

### motion/ — change
- `kinematics.py`: absorb `position_from_well_id` / `well_id_from_position`; add `well_to_relative_steps(from_well, to_well, plate_type)` returning `(dx, dy, dz)` for direct `M` streaming (Y-only — fixes B4 permanently, no more `A` command in any live path).
- `trajectory.py`: `plan_routine(wells, plate_type)` must emit the move list the runner streams — confirm output shape is `[{dx,dy,dz,well_id,params}, ...]`; add `dwell`/capture markers so the runner knows where to stop.

## 7. BUILD ORDER & ACCEPTANCE TESTS

**Order:** §3 backend runner (unblocks everything) → §4 flash v2 → cronjob HTTP switch →
§5.1 mount V2 + Run/Stop → §5.4 progress/toasts → §5.2–5.3 style migration → §6 deletions.

Acceptance (each maps to a finding):
- **N2/N1**: start a routine from UI while clicking manual X-jog → jog returns 409, routine unaffected.
- **N3**: ProgressBar advances well-by-well during a live routine.
- **N4**: Stop mid-move → stage halts < 1 s, motors disabled, progress shows `aborted`.
- **B3**: well with `lightTime = 500` → GPIO high for 0.5 s (scope or log timestamps).
- **B4**: row change moves Y only; Z position in `?` report unchanged.
- **B5**: unplug USB mid-routine → routine errors out with toast, no silent position drift.
- **B10**: 48-well plate in V2 — selecting well F8 edits F8.
- Cron: timer fires → `/api/routine/run` logged, routine completes, pictures saved.
